### 1.2 持久記憶功能 / Agent 及排程長記憶方案

- **目標**：實現跨 Session 的長期記憶，讓 Agent 能累積知識與追蹤長期趨勢
- **優先級**：P1（短期～中期）
- **關聯**：`digest-memory` Skill、`context/continuity/` 連續記憶機制、知識庫筆記 `b1640648`（AI Agent 記憶系統研究）、筆記 `1b5eb493`（Mem0/Letta/MemGPT 指南）
- **現況**：
  - 已有 `digest-memory.json`（跨次摘要記憶）和 `context/continuity/auto-task-*.json`（自動任務連續記憶）
  - 但缺乏**語義記憶整合**（KB 知識與 Agent 決策的橋樑）和**記憶衰減機制**
- **KB 研究基礎**：
  - 三層記憶模型（感覺/短期/長期）已在 KB 有深度研究
  - StateLM（主動記憶管理）、Chain-of-Memory（輕量鏈式記憶）值得借鑑
  - Continuum Memory Architecture（CMA）適合本專案的連續執行模式
- **可執行動作**：
  1. 設計「記憶整合層」：自動將 `continuity/*.json` 的重複模式提煉到 KB 筆記（類似 Memory Consolidation）
  2. 為 `digest-memory.json` 加入趨勢摘要欄位（7 日/30 日滑動視窗）
  3. 評估 Mem0 或 Letta 作為外部記憶引擎的可行性（對比現有 JSON 方案的擴展性）

## 二、技術棧擴充

### 2.1 Hono — 超輕量 Edge-First Web 框架

- **目標**：評估 Hono 作為 `bot/groq-relay.js`（Express）替代方案的可行性
- **優先級**：P2（中期）
- **關聯**：`bot/groq-relay.js`（現用 Express）、`config/llm-router.yaml`、KB 筆記 `7d524b1e`（Hono 深度研究）
- **KB 研究基礎**：
  - Hono v4.12.5，29.2K Stars，基於 Web Standards（Request/Response/fetch API）
  - RegExpRouter 單一正則路由引擎，極低延遲
  - 支援 Cloudflare Workers、Deno、Bun、Node.js 等 8+ Runtime
- **效益評估**：
  - 若遷移到 Cloudflare Workers 部署，可降低本機依賴、提升可用性
  - RPC 端對端型別安全可改善 `groq-relay` 的 API 契約
- **可執行動作**：
  1. 建立 `bot/groq-relay-hono.js` 原型，用 Hono 重寫核心 3 個端點
  2. 效能對比測試：Express vs Hono（延遲、記憶體、cold start）
  3. 評估 Cloudflare Workers 部署方案（搭配 D1 資料庫取代本地 JSON）

### 2.2 加入 LlamaIndex + Mistral 3.2 Small

- **目標**：引入 LlamaIndex 作為 RAG 框架 + Mistral 3.2 Small 作為輕量推理模型
- **優先級**：P2（中期）
- **關聯**：`config/llm-router.yaml`（現有 Groq/Claude 雙軌）、`skills/groq/SKILL.md`、KB 筆記（AI Agent 框架比較）
- **現況**：
  - LLM 路由已有 mapping 格式（O(1) lookup），新增模型只需擴充 `routing_rules`
  - Groq 處理輕量任務（摘要/翻譯/分類），Claude 處理複雜任務
- **效益評估**：
  - LlamaIndex 的 RAG 管線比手工 KB 查詢更結構化（自動 chunking、reranking）
  - Mistral 3.2 Small 適合本地部署輕量推理，降低 API 成本
- **可執行動作**：
  1. 在 `config/llm-router.yaml` 新增 `mistral_small` 路由規則（task_type: `local_inference`）
  2. 建立 `tools/llamaindex_rag.py` 原型：串接 KB SQLite + FTS5 + 向量索引
  3. 效益量化：比較 LlamaIndex RAG vs 現有 hybrid search 的召回率與精確度

### 2.3 Langfuse — LLM 可觀測性平台

- **目標**：評估 Langfuse 作為 Agent 可觀測性補充方案
- **優先級**：P3（長期）
- **關聯**：來源連結 `https://langfuse.com/blog/2025-03-19-ai-agent-comparison`、KB 筆記 `c1b8729f`（Langfuse 深度洞察）、`tools/trace_analyzer.py`（現有追蹤）
- **現況**：已有 `post_tool_logger.py`（JSONL 日誌）+ `trace_analyzer.py`（規則式根因分析）
- **KB 研究基礎**：
  - Langfuse 23K Stars，佇列式攝入架構
  - LLM-as-a-Judge 雙流水線評估
  - Prompt 版本管理與 A/B 測試
- **可執行動作**：
  1. 閱讀 Langfuse blog 文章，整理 AI Agent 框架比較重點
  2. 評估 Langfuse 自架（Docker）vs 雲端版的成本與複雜度
  3. 設計 `post_tool_logger.py` → Langfuse 的資料橋接方案（JSONL → OpenTelemetry）
