## 一、系統穩定性（P0 — 立即影響執行品質）

> 本區項目直接影響日常自動化的成功率與資源效率，應優先處理。

### 1.1 自動任務公平輪轉修復 🔴

- **目標**：解決 auto_task_fairness=1.993（Gini 係數）、29 個任務僅 6 個有執行紀錄的嚴重失衡
- **優先級**：P0（短期，ADR-20260318-027 Proposed/pending）
- **關聯**：`task-fairness-analyzer` Skill（#40）、`run-todoist-agent-team.ps1`、`context/auto-tasks-today.json`
- **Done 條件**：7 天後 executed task 覆蓋率 ≥60%、idle 任務數 ≤5
- **執行計畫**（來自 improvement-backlog）：
  1. 盤點近 7 天各 task_key 的執行次數、連續跳過次數與最近執行時間，建立 starvation score
  2. 在 Phase 1 選任務前加入 weighted round-robin 規則，優先補償長期未執行任務
  3. 把選任務理由寫入 state（selected、skipped、skip_reason、starvation score）
  4. 新增 7 天公平性驗證腳本或檢查點
- **可執行動作**：
  1. 用 `task-fairness-analyzer` Skill 產出當前公平性報告作為 baseline
  2. 修改 `run-todoist-agent-team.ps1` Phase 1 加入 starvation-aware 選擇邏輯
  3. 在 `check-health.ps1` 加入公平性指標（Gini < 1.0 為健康）

### 1.2 Phase 2 快照恢復與故障續跑 🔴

- **目標**：Phase 2 Agent 超時或失敗時，已完成的結果可被快照保留，下次僅重試失敗項
- **優先級**：P0（短期，ADR-20260318-028 Proposed/pending，直接降低資源浪費）
- **關聯**：LangGraph 持久化執行模式、`state/run-fsm.json`、`run-todoist-agent-team.ps1`
- **Done 條件**：Phase 2 部分失敗後可從 snapshot 續跑，不重複已成功 Agent
- **現況痛點**：近 7 天成功率 85.1%，任一 Phase 2 Agent 超時 → 整個 Phase 2 重跑
- **執行計畫**（來自 improvement-backlog）：
  1. 定義 `state/phase2-snapshot-{traceId}.json` schema（trace_id、已完成/失敗 agents、checkpoint 時間、可重用 result paths）
  2. Phase 2 每次 Agent 完成或失敗時即時落盤 snapshot
  3. 重試/下次執行先讀 snapshot 決定是否跳過已完成工作
  4. Resume 決策綁定 run-fsm 狀態，避免殭屍 run 或過期 snapshot 被誤用
- **可執行動作**：
  1. 設計 snapshot JSON Schema，放入 `config/schemas/`
  2. 在 `run-todoist-agent-team.ps1` Phase 2 迴圈加入 per-agent checkpoint 邏輯

### 2.2 Skill 分層管理（三層 Skill 系統）

- **目標**：引入 core → shared → workspace 三層 Skill 分層機制，提升生命週期管理
- **優先級**：P1（短期～中期）
- **關聯**：improvement-backlog 提案（源自 OpenClaw 302K Stars）、`skills/SKILL_INDEX.md`、`docs/ARCHITECTURE.md`
- **來源靈感**：OpenClaw 的 bundled → managed → workspace 三層 Skill 系統
- **現況分析**：
  - 48 個 Skills 全部扁平管理，無穩定度分層
  - 部分實驗性 Skill（如 skill-forge、workflow-forge）與核心 Skill（todoist、ntfy-notify）同層級
- **建議分層**：
  - **core**（~15 個）：每日摘要必用、高穩定度（todoist、pingtung-news、ntfy-notify、api-cache 等）
  - **shared**（~10 個）：跨專案共用、中等穩定度（knowledge-query、web-research、groq 等）
  - **workspace**（~23 個）：實驗性或低頻使用（skill-forge、workflow-forge、behavior-pattern-analyzer 等）
- **可執行動作**：
  1. 為 `skills/SKILL_INDEX.md` 加入分層標記（`tier: core|shared|workspace`）
  2. 建立 machine-readable 的 `skills/manifest.json`（參考 n-skills 的 openskills 格式），降低文字索引的路由不精確性
  3. 制定分層升降規則：workspace Skill 累積 30+ 次成功執行後可升級為 shared

### 2.3 持久記憶功能 / Agent 及排程長記憶方案

- **目標**：實現跨 Session 的長期記憶，讓 Agent 能累積知識與追蹤長期趨勢
- **優先級**：P1（短期～中期）
- **關聯**：`digest-memory` Skill、`context/continuity/` 連續記憶機制、KB 筆記 `b1640648`（AI Agent 記憶系統研究）、`1b5eb493`（Mem0/Letta/MemGPT 指南）、`72c33b6d`（Memory-Augmented LLM 指南）、`1adc3ffd`（Hindsight 四網路記憶）、`14c8dbd1`（Mastra Observational Memory）
- **現況**：
  - 已有 `digest-memory.json`（跨次摘要記憶）和 `context/continuity/auto-task-*.json`（自動任務連續記憶）
  - 缺乏**語義記憶整合**（KB 知識與 Agent 決策的橋樑）和**記憶衰減機制**
- **KB 研究基礎（2026-03 最新）**：
  - 三層記憶模型（感覺/短期/長期）已在 KB 有深度研究
  - StateLM（主動記憶管理）、Chain-of-Memory（輕量鏈式記憶）值得借鑑
  - **Hindsight**（vectorize-io, 4.3K→4.6K Stars）：四網路記憶 + Retain-Recall-Reflect + MCP 原生整合，LongMemEval 91.4% SOTA
  - **Mastra**（21.8K Stars）：Observational Memory 三層壓縮，LongMemEval 94.87%
  - **Mem0** 效能數據：26% 準確率提升、91% 延遲降低、90% Token 節省
- **ROI 評估框架**：

  | 方案 | 整合難度 | 本專案適配度 | 維護成本 |
  |------|---------|-------------|---------|
  | 現有 JSON 方案 | 已完成 | 高（自建） | 低 |
  | Mem0/Mem0ᵍ | 中等 | 高（跨 Session） | 中 |
  | Hindsight + MCP | 中等 | **最高**（MCP 原生） | 低 |
  | Mastra Observer | 高 | 中（TypeScript） | 中 |
  | MemGPT/Letta | 高 | 中（需 Docker） | 高 |

- **可執行動作**：
  1. 設計「記憶整合層」：自動將 `continuity/*.json` 的重複模式提煉到 KB 筆記
  2. 為 `digest-memory.json` 加入趨勢摘要欄位（7 日/30 日滑動視窗）
  3. **優先評估 Hindsight + MCP**：因 MCP 原生整合與本專案架構最契合，且不需 Docker
  4. 建立記憶品質指標：recall accuracy、freshness score、dedup ratio
